## Project Philosophy

**Primary Motivation:** Teaching obligation - sharing knowledge across formats (digital, in-person, written/published). Not profit-driven. The goal is helping everyone with even small interest in modular synthesis understand it from the ground up.

**Core Principle:** Teach interconnection, not procedures. Show WHY and HOW things work, not just WHAT buttons to press.

**Target Outcome:** Being known as someone who helps. Ground-up modular education that's accessible to absolute beginners.

---

## Module Selection Criteria

**Established Manufacturers Focus:**
- Doepfer (foundational, widely available)
- Make Noise (innovative, well-documented)
- Erica Synths (quality, accessible)
- Intellijel (professional grade)
- Mutable Instruments (even though discontinued - too important to skip)
- Other established brands with longevity

**Selection Principles:**
- Must be foundational pieces that teach transferable concepts
- Widely available with long-term market presence
- Skip tiny one-person operations (except notable exceptions)
- Include important discontinued modules (Plaits, Rings, etc.) - legacy matters

**Categories to Cover:**
- VCOs (voltage-controlled oscillators)
- Filters (VCFs)
- VCAs (voltage-controlled amplifiers)
- Envelope Generators
- LFOs and Modulation Sources
- Maths (non-negotiable - too foundational)
- Sequencers
- Effects
- Utilities

**Format Pattern:**
- Module Type (general function explanation)
- Module Example (specific implementation)
- Notable Mentions (discontinued but important alternatives)

Example: "VCOs work like this [principle], here's Plaits as an example, also consider Instruo Cs-L [alternative]"

## Diagram Standards

**Approach:** Top-down case view with simplified module representations

**Tools:**
- Primary: draw.io or Affinity Designer (vector/layout software)
- Production-quality diagrams, not sketches

**Visual Design:**
- Simplified module boxes (white background, approximate jack locations)
- NO photo overlays (too messy with cable routing)
- Clear jack labeling
- Color coding system:
  - CV = [color TBD - maybe blue]
  - Audio = [color TBD - maybe red/orange]
  - Gate/Trigger = [color TBD - maybe yellow/green]

**Routing Clarity:**
- Explicit "Jack A on Module X → Jack B on Module Y" instructions
- Visual lines showing cable paths
- Numbered/lettered steps for complex patches

**To Develop:**
- Template for standard module representations
- Reusable component library
- Consistent sizing/spacing standards

## Content Structure & Complexity

**Progressive Learning Model:**
- Start as simply as possible
- Every page assumes ONLY what came before
- Continuously build: terms → patches → principles
- No knowledge assumptions beyond what we've taught

**Teaching Pattern: Evergreen + Concrete**

Format for each concept:
1. **General Principle** - "VCO → Filter → VCA → Effects" (signal flow concept)
2. **Specific Example** - "Castor & Pollux → Forbidden Planet → Black VCA2 → Ghost" (concrete implementation)

This dual approach teaches:
- WHY it works (transferable understanding)
- HOW to implement it (practical application)

**Acknowledge Change:**
- Markets evolve, modules get discontinued
- Focus on principles that outlast specific products
- Plan for periodic updates (see Maintenance Strategy)

## Book Structure (Proposed)

**Working Length:** 250-300 pages

### Part 1: Before You Start (40-50 pages)
- 00: Before You Buy Anything ✓ (already written)
- 01: Making Sound (in progress)
- 02: Controlling Sound
- 03: Musical Phrases
- 04: Rhythm and Percussion

### Part 2: Understanding Interconnection (100-120 pages)
- Philosophy chapter: WHY interconnection matters
- 15-20 selected modules as teaching examples
- Each module chapter structure:
  - What it does
  - Why it works this way
  - How it connects to other modules
  - What it teaches about synthesis broadly
- NOT comprehensive coverage - selective teaching examples
- Mix of categories (oscillators, filters, modulators, effects, utilities, sequencers)

### Part 3: Patching Studies (60-80 pages)
- Progressive complexity patches with full diagrams
- Each patch teaches a specific principle
- Demonstrate: SAME sequence through different processing = different results
- Show how changing one connection transforms output

### Part 4: System Thinking (30-40 pages)
- Case study: building a complete system
- Possible focus: 280HP travel case as working example
- Power management strategies
- Module selection rationale
- Addressing common gaps
- System evolution over time

### Part 5: Reference (20-30 pages)
- Glossary of terms
- Troubleshooting common issues
- Resources for continued learning
- Quick-reference tables
- Index

## Economics & Distribution

**Pricing Philosophy:**
- NOT profit-driven - teaching obligation is primary motivation
- Break-even model: cover printing, shipping, tools/software/hosting
- No subsidy from author, but no profit extraction either

**Target Price Range:** $30-45 USD
- Comparable music books (Patch & Tweak) run $75+
- Want accessibility without asking author to subsidize
- Final pricing depends on page count and production costs

**Distribution Options:**
- Amazon KDP (Kindle Direct Publishing) - print-on-demand
  - No upfront costs
  - No inventory burden
  - ~60% royalty after printing costs
  - Wide distribution

**Format Options to Consider:**
- Print-only (physical reference for patching)
- Print + bundled PDF (buy one, get both)
- Separate digital PDF (Gumroad, own site) for lower price point

**Status:** Distribution strategy to be finalized closer to completion

## Maintenance Strategy

**Update Cadence:** Every 3-4 years

**Rationale:**
- Modular market moves fast (new modules, discontinued classics)
- Principles remain evergreen, but examples age
- Regular updates maintain relevance without constant churn

**Version Management:**
- Each edition clearly dated
- Maintain online errata/supplement between editions
- GitHub documentation remains living/current reference
- Book is snapshot of knowledge at publication date

**Approach:**
- Focus on evergreen principles over specific products where possible
- When citing specific modules, acknowledge potential obsolescence
- Structure content so core teaching survives product changes
